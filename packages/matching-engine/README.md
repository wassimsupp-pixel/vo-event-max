No generative AI, no machine learning ‚Äî every decision is explainable from a fixed scoring table.

## Scoring table

| Signal | Points | Key |
|---|---|---|
| Email exact normalised match | **+100** | `email_exact` |
| ID exact match (both non-empty) | **+90** | `id_exact` |
| Full name exact normalised match | **+80** | `name_exact` |
| Fuzzy full name (`token_sort_ratio √ó 0.75`) | **‚â§ 75** | `name_fuzzy` |
| Company normalised bonus | **+5** | `company_bonus` |
| Phone normalised bonus | **+5** | `phone_bonus` |

Bonus signals are **always** evaluated and stack on top of the best primary signal.  
Scores above 100 are possible (e.g. email_exact + both bonuses = 110).

---

## Decision thresholds

| Decision | Score range | Recommended action |
|---|---|---|
| `CERTAIN` | ‚â• 90 | Auto-merge records |
| `PROBABLE` | 70 ‚Äì 89 | Merge + flag for a quick human check |
| `TO_VERIFY` | 50 ‚Äì 69 | Route to human review queue |
| `NOT_FOUND` | < 50 | Create exception / manual lookup |

Thresholds are deliberately **conservative**: it is always safer to flag a record than to silently merge two unrelated participants.

---

## Normalisation rules

| Field | Rule |
|---|---|
| `email` | `strip()` + `lower()` |
| `name` (first or last) | NFD decompose ‚Üí strip Non-spacing Marks ‚Üí ASCII ‚Üí `lower().strip()` |
| `phone` | Strip all non-digits |
| `flight_number` | `upper()` + strip all whitespace |
| `date` | Try five formats in order: `%d/%m/%Y`, `%Y-%m-%d`, `%d-%m-%Y`, `%m/%d/%Y`, `%d.%m.%Y` |

## Using the engine in code

```python
from matcher import ParticipantRecord, match_sources, detect_duplicate_emails

# Build records from your data source
registrations = [
    ParticipantRecord(
        id="REG001",
        first_name="Jean-Pierre",
        last_name="De Smedt",
        email="jp.desmedt@corp.be",
        company="Solvay SA",
        phone="+3222345678",
        source="registration_sheet",
    )
]

flight_records = [
    ParticipantRecord(
        id="FCM001",
        first_name="Jean Pierre",     # hyphen dropped -- noise
        last_name="Desmedt",          # space merged -- noise
        email="jp.desmedt@corp.be",   # email still matches
        source="fcm_export",
    )
]

results = match_sources(registrations, flight_records)

for r in results:
    print(f"{r.registration.id} -> {r.decision.value}  score={r.score:.1f}")
    print(f"  signals: {r.signals}")
    if r.conflict_fields:
        print(f"  conflicts: {r.conflict_fields}")

# Duplicate email detection
duplicates = detect_duplicate_emails(registrations)
for group in duplicates:
    print("Duplicate group:", [rec.email for rec in group])
```

Expected output:

```
REG001 -> certain  score=100.0
  signals: {'email_exact': 100.0}
```

## Design decisions

* **Email is the gold key.** A verified email match scores 100 and alone triggers `CERTAIN`.  
  This reflects the real-world fact that email addresses are system-of-record identifiers in most conference management platforms.

* **Name fuzzy matching is capped at 75.** This means a fuzzy-only match can at most reach `PROBABLE` (70‚Äì89 range with bonuses).  
  A perfect name match scores 80 (enough for `CERTAIN`), while a degraded fuzzy match needs a bonus to cross 70.

* **Bonus signals never unlock a false merge.** Company and phone signals add at most 10 points combined ‚Äî they cannot turn a low-confidence name match into `CERTAIN`.

* **Conflicts are surfaced, not suppressed.** Even on a `CERTAIN` match, if the email in the FCM record differs from the registration email, that conflict is reported in `conflict_fields` so a human can investigate without having to re-run the match.
